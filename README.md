# 📅 数理最適化タスクスケジューラー

このプロジェクトは、数理最適化（混合整数計画問題：MIP）を用いて、個人の稼働可能時間（キャパシティ）とタスクの締め切り・優先度を考慮し、日別の最適なTodoリストを自動生成するシステムです。

---

## 📂 フォルダ構成

* **`todos/`**: プロジェクトやテーマごとのタスク管理Markdownファイルを格納します（ここに手動でタスクを書き込みます）。
* **`dailynotes/`**: 自動生成される日別のTodoリスト（デイリーノート）です。
* **`deadlines/`**: 自動生成される締め切り・実行計画一覧（`deadlines.md`）が格納されます。
* **`analysis/`**: 自動生成されるリソース過不足分析レポート（`allocation_report.md`）および空き時間分析レポートとグラフ（`free_time_report.md`, `free_time_chart.png`）が格納されます。
* **`src/`**: 最適化スクリプト（`optimize_schedule.py`）や空き時間可視化スクリプト（`generate_free_time_chart.py`）が格納されています。

---

## 📝 タスクの記載方法（`todos/` 内のファイル）

各プロジェクトのMarkdownファイル（例: `todos/project_a.md`）には、以下の書式で未完了タスク（`- [ ]`）を記述します。

### 基本書式
```markdown
- [ ] タスク内容 [優先度] [📅 締め切り] [⏳ 見積もり時間]
```

### パラメータの書き方

1. **見積もり時間 (`⏳`)**:
   - `⏳ 2h`（2時間）や `⏳ 30m`（30分）のように記述します。
   - **未指定の場合**: デフォルトで `⏳ 1h` として計算されます。

2. **優先度 (`p3` / `p2` / `p1`)**:
   - `p3`：高（優先して早い日にスケジュールされます）
   - `p2`：中（デフォルト。記述を省略できます）
   - `p1`：低（後回しが許容されます）
   - ※キーボードで `p3` または `p1` と入力するだけで認識されます（`p2` は省略可）。

3. **日付・締め切り (`📅`)**:
   - **日付がすでに書かれているタスク (`📅 YYYY-MM-DD`)**:
     - 「日付固定タスク」として扱われます。ソルバーは**必ずこの日付以前**に割り当てようとします（ハード制約）。
     - キャパシティ不足でどうしても間に合わない場合のみ、締め切りをオーバーして配置され、ボトルネック分析に警告が表示されます。
   - **日付が書かれていないタスク (日付なし)**:
     - 「自由タスク」として扱われます。全体のキャパシティ制限を守りながら、優先度順に最適な空き日に自動でスケジュールされます。
     - **スケジュールされた日付は、自動的に元のマークダウンファイルに書き戻されます。**

4. **階層化（ネスト）されたタスクのルール**:
   - **集計用属性（`📅`, `⏳`, `p3`/`p1`）は、必ず「1番ネストが深い末端のタスク」にのみ記述してください。**
   - 子タスク（インデントが深い行）を持つ親タスク行には、日付や工数は記述しません。
   - ソルバーは、末端の未完了タスクのみをスケジュール対象として抽出するため、時間が重複してカウントされるのを防ぎます。自動配分時の日付の書き戻しも、末端タスクに対してのみ行われます。

#### 記載例：
```markdown
# 新規プロジェクトの準備

- [ ] 資料作成                         <!-- 親タスク：属性は書かない -->
  - [ ] 企画書スライドの執筆 p3 ⏳ 2h   <!-- 末端タスク：属性を記述（日付未定なので自動配分対象） -->
  - [ ] マニュアルの作成 📅 2026-07-10 ⏳ 1h  <!-- 末端タスク：属性を記述（日付固定） -->
- [ ] 関連資料の調査 p1                 <!-- 子タスクを持たないため、これ自体が末端タスク -->
```

## ⚙️ 実行方法 (Docker Compose)

ホストマシンの環境を汚さずに、Dockerコンテナ内で最適化ソルバーを実行します。

### 1. スケジュールの自動最適化を実行する
プロジェクトのルートディレクトリで以下のコマンドを実行します。

```bash
docker compose up
```

- 初回実行時、または `Dockerfile` や `docker-compose.yml` を書き換えた場合は、以下のコマンドでビルドを伴って実行してください：
  ```bash
  docker compose up --build
  ```

### 2. 実行後の変化
- **プロジェクトファイル (`todos/*.md`)**: 日付のなかったタスクに、自動決定された日付 `📅 YYYY-MM-DD` が追記されます。
- **日別ノート (`dailynotes/YYYY-MM-DD.md`)**: その日に実行すべきタスクが自動転記されて生成されます。
- **計画・ボトルネック分析 (`deadlines/deadlines.md`)**: 全体の時系列計画表と、キャパシティが不足している期間のボトルネック分析レポートが作成されます。
- **リソース・空き時間分析 (`analysis/`)**:
  - `allocation_report.md`: 期限付きタスクとリソース過不足の時系列分析レポート（過不足状況の具体的な原因タスク一覧も掲載されます）。
  - `free_time_report.md`: 今後30日間の空き時間（バッファ）に関する週次・日次レポート。
  - `free_time_chart.png`: 割り当て済みタスク工数と空き時間の推移を可視化した積層棒グラフ。

## 📊 キャパシティとボトルネック分析について

### 曜日ごとのキャパシティ（初期設定）
- **平日（月〜金）**: 最大 **2.0時間** (120分) / 日
- **休日（土・日）**: 最大 **6.0時間** (360分) / 日

### ⚠️ 物理的キャパシティボトルネック分析の読み方
すべての締め切りタスクの総見積もり時間が、締め切り日までの利用可能時間を上回っている場合、[deadlines/deadlines.md](file:///Users/masashiueno/Library/Mobile%20Documents/iCloud~md~obsidian/Documents/Chokosenloveti-mac/deadlines/deadlines.md) の末尾にボトルネックレポートが出力されます。

```markdown
| 判定基準日 | 累積利用可能時間 | 同日までの締切タスク合計時間 | 過不足状況 |
| --- | --- | --- | --- |
| 2026-07-06 | 2.0h | 5.0h | ❌ 不足 (3.0時間不足) |
```

- **累積利用可能時間**: 開始日からその基準日までに作業できる合計時間。
- **同日までの締切タスク合計時間**: 同日までに締め切りが設定されている全タスクの合計時間。
- **過不足状況**: マイナスになっている場合、その期間で「物理的に稼働時間が足りていない（計画が破綻している）」ことを示します。タスクの締め切りを延ばすか、見積もり時間を調整する判断材料にしてください。
